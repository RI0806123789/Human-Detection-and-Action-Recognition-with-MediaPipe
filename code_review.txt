train.py コードレビュー
🔴 バグ（実行時エラーになるもの）
1. plot_training_history でKeyErrorが発生
最終モデルの学習時に validation_data を渡していないのに、plot_training_history 内で history.history['val_accuracy'] と history.history['val_loss'] を参照している。これは必ずクラッシュする。
2. split_dataset の n_train < 0 チェックが死にコード
int() は切り捨て除算なので n_test + n_val が n を超えることはなく、n_train < 0 になる条件は数学的に存在しない。意図した防御コードが機能していない。
3. サンプル数が少ないクラスで val/test が 0件になる
例えば1クラス4枚の場合、int(4 * 0.2) = 0 になりval・testに1枚も割り当てられない。そのクラスのメトリクス計算が無意味になるが、エラーも警告も出ない。

🟠 モデル設計の問題
4. Conv2D が1層しかない
128×128の画像に対してConv2D×1+MaxPooling×1だけでは特徴抽出が極めて浅い。精度に直結する根本的な問題。
5. Flatten 後のノード数が膨大
128×128→Conv(3×3)→126×126×32→MaxPool→63×63×32→Flatten で 127,008ノード。その後のDense層のパラメータ数が爆発する。GlobalAveragePooling2Dを使えば大幅に削減できる。
6. BatchNormalization がない
学習の安定性・収束速度が低下する。Optunaの多試行でより顕著になる。
7. 転移学習を使っていない
小規模データセットに見えるが、EfficientNetやMobileNetなどの事前学習モデルを使えば精度が大幅に改善できる可能性が高い。
8. Optunaがチューニングするのは全結合層のみ
Conv層の数・フィルター数・カーネルサイズ・L2正則化などがすべて固定。アーキテクチャ探索になっていない。

🟠 学習プロセスの問題
9. 最終モデルに EarlyStopping がない
Optunaの各試行ではEarlyStoppingを使っているのに、最終学習では使っていない。MAX_EPOCHSまで必ず回り過学習を検知できない。
10. 最終モデルに validation_data がない
過学習を監視する手段がゼロ。学習曲線のval側も表示できないため、上記バグ①の原因にもなっている。
11. グローバルなTFシードが設定されていない
tf.random.set_seed() がないため、同じOptunaシードでも結果が再現されない。
12. Optunaの試行中に verbose=1 のまま
試行数が多いと出力が膨大になる。verbose=0 にすべき。
13. 学習率スケジューリングがない
固定学習率のみ。ReduceLROnPlateauやCosineDecayなどを探索対象にすれば精度改善の余地がある。

🟠 Optuna設定の問題
14. Prunerが設定されていない
MedianPruner や HyperbandPruner を使えば明らかに低性能な試行を早期打ち切りできる。
15. Studyに永続化ストレージがない
中断した場合に試行結果が消える。storage="sqlite:///optuna.db" などを指定すべき。
16. 毎試行でデータセットを再作成している
batch_sizeが同じ試行でもデータセットを毎回作り直している。batch_sizeが変わる場合のみ再作成すれば良い。

🟡 データパイプラインの問題
17. .cache() がない
毎エポック、毎試行でディスクから画像を再読み込みしている。map().cache().batch() の順にすれば2エポック目以降はメモリから読める。
18. decode_image 後に set_shape() がない
tf.image.decode_image は静的なshapeを返さないため、tf.dataのグラフ内でshape情報が (None, None, None) になる可能性がある。
19. アスペクト比を無視したリサイズ
tf.image.resize は単純に変形するため、縦長・横長画像が歪む。resize_with_crop_or_pad や resize_with_pad の検討が必要。
20. データ拡張が最小限すぎる
RandomFlip(水平)とRandomZoom(±5%)のみ。回転・輝度・コントラスト・ShiftなどはOptunaで探索する価値がある。特にRandomZoom(0.05)はほぼ無効に等しい。
21. 破損ファイルのエラーハンドリングがない
読み込みに失敗した画像で学習全体がクラッシュする。

🟡 コード品質の問題
22. create_dataset が死んだコードとして残っている
docstringで「Deprecated」と書かれているが削除されていない。混乱の元。
23. main() が長すぎる
1関数が全工程を担っている。run_optuna()・train_final_model()・evaluate() などに分割すべき。
24. 型ヒントがない
関数の引数・戻り値の型が不明で可読性・保守性が低い。
25. ロギングフレームワークを使っていない
すべて print() で、デバッグレベルの切り替えや出力先変更ができない。
26. Configの値をバリデーションしていない
config.n_trials が0以下だった場合など、実行して初めてエラーになる。
27. クラス名とラベルのマッピングを保存していない
class_names がモデルと別々に保存されているため、推論時に対応が取れなくなるリスクがある。
28. plt.show() を本番コードで呼んでいる
サーバー環境や自動実行環境でブロックする可能性がある。matplotlib.use('Agg') の指定や show() の条件分岐が必要。

🟡 評価・メトリクスの問題
29. 混同行列の可視化がない
クラス間の誤分類パターンが把握できない。分類タスクで最も有用な可視化の1つが欠けている。
30. model.evaluate() を呼んでいない
最終的なテスト精度・損失を1行で得られるのに、model.predict() から手計算している。
31. 各Splitのサンプル数・クラス分布を出力していない
train/val/testに何枚ずつ入ったか確認できない。
32. クラス不均衡への対処がない
class_weight の計算・指定がないため、サンプル数の多いクラスに偏った学習になる可能性がある。

🟡 その他
33. collect_image_paths の空ディレクトリ対応がない
画像のないクラスフォルダが存在するとラベルだけ割り当てられる。
34. Datasetディレクトリ不在のエラーハンドリングがない
dataset_dir が存在しない場合、エラーメッセージが不親切。
35. モデルの保存先がscriptのディレクトリ
Graph/ に保存されるグラフと、scriptと同階層に保存されるモデルで保存場所が統一されていない。
36. collect_image_paths のクラス名ソート順に依存している
ディレクトリ名の変更・追加でラベルマッピングが変わるが、それが記録されない。
